# Personal OS - Database Schema Specification V1

This schema defines all core entities and relationships required for Personal OS.

The schema should prioritize:

* Simplicity
* Extensibility
* Automatic date tracking
* Relationship support
* Analytics generation

Dashboard tasks should come from:

WHERE

completed = false

ORDER BY

scheduled_time ASC

alphabetical order

# Version 1 Rule

Keep the schema relational.

Do not introduce:

JSON blobs

Dynamic properties

Custom databases

Plugin schemas

User-generated table structures

The database should remain simple, explicit, and optimized for personal productivity tracking.

# Database Design Notes

## Progress Fields

Do not permanently store progress values in the database.

The following fields should be treated as computed values:

* Skill Progress
* Project Progress
* Goal Progress

Progress should be calculated dynamically from:

Checklist Completion

Linked Item Completion

Goal Progress Rules

This prevents data inconsistencies.

---

## Checklist Relationships

Avoid polymorphic references such as:

entity_type

entity_id

for checklist ownership.

Instead use proper relationship tables:

skill_checklists

project_checklists

goal_checklists

workout_checklists

linked to a shared:

checklists

table.

This preserves database integrity while allowing a reusable checklist system.

---

## Dashboard Queue

The Dashboard queue should be generated directly from:

checklist_items

where:

completed = false

Sorting Rules:

1. Scheduled Tasks

   * Sort by scheduled_date
   * Then scheduled_time

2. Unscheduled Tasks

   * Sort alphabetically

3. Completed Tasks

   * Display separately in a Completed Today section

---

## Life Area Integration

The following entities must support a Life Area relationship:

* Skills
* Projects
* Goals
* Learning Sessions (inherit from Skill)
* Health Goals
* Fitness Goals

Life Areas are used throughout analytics, dashboard summaries, and reporting.

---

## Automatic Date Rules

All major entities should automatically manage:

created_at

updated_at

started_at (when applicable)

completed_at (when applicable)

Users should never manually manage system timestamps.

---

## Database Philosophy

The schema should remain:

* Relational
* Explicit
* Predictable
* Analytics-Friendly

Avoid:

* Dynamic schemas
* Custom database builders
* Plugin-driven tables
* User-generated table structures
* Notion-style property systems

Personal OS is a purpose-built personal operating system, not a general-purpose workspace platform.
